Gate logic distilled

The important behavior is less “two big flowcharts” and more these invariants.

1. Bootstrap is a one-shot handoff

The interactive run creates the cron continuation, records concrete bootstrap config, marks any explicit CRON bootstrap row done, commits the plan update, reports the handoff, then stops.

2. Cron consumes bootstrap state

Each cron tick starts in a fresh session, reads the plan/config, requires any explicit CRON bootstrap row to already be done, and then follows the configured continuation loop.

3. Completion is verified every tick

After a phase attempt, failure, or no eligible work, the tick re-reads the plan, runs safe verification, checks git status, and only pauses/removes its own cron job when complete.

Bootstrap and continuation responsibilities

Bootstrap state is represented by ## CRON Bootstrap and any explicit CRON bootstrap status row.

Interactive run

  • ✓May create the cron job with a self-contained prompt.
  • ✓Records ## CRON Bootstrap, marks only an explicit bootstrap row done, and commits the scoped plan update.
  • ✓Reports the job id/name/schedule and stops; the initiating agent is done.

Cron tick

  • ✓Consumes the recorded bootstrap state and owns scheduled continuation work.
  • ✓Starts with the first eligible TODO implementation phase according to the plan.
  • ✓Self-pauses by default, or removes itself only when explicitly configured.
