A self-contained visual guide to the decision gates, durable plan surface, branch/worktree rules, phase execution loop, cron bootstrap path, and completion verifier described in SKILL.md.

Operating thesis

The existing plan file is the durable control surface. Git provides local coordination through explicit branches, optional worktrees, scoped checkpoint commits, and local verification. Chat memory and remote PR/CI automation are intentionally outside the default workflow.

  • 1Require an existing plan file and discover repository state before execution; never assume the base branch.
  • 2Drive work from the bottom ## Phase Status table and update it according to allowed transitions.
  • 3In parallel mode, isolate each phase by branch and worktree; do not race on the canonical status table.
  • !Stop before unrelated changes, destructive cleanup, unauthorized merges, ambiguous ownership, or failed verification.

Default boundaries

Local only. Excludes PR creation, CI polling, merge automation, broad cleanup, destructive git actions, and worktree removal unless separately authorized.

End-to-end logic flow

Diagram grounded in the repository's SKILL.md: Existing Plan Requirements, Execution Workflow, Branch and Worktree Rules, Cron Scheduling, Completion / Signoff Verifier, and Stop Conditions.

Existing plan intake

1

Confirm the plan path, repository path, execution mode, and slug from the existing file.

2

Run preflight: git rev-parse, current branch, status, worktree list.

3

Validate branch strategy, worktree strategy, phase scopes, dependencies, verification commands, exclusions, and stop conditions.

4

Require the canonical ## Phase Status table before selecting work.

Execution branch

1

Re-read the plan from disk and parse statuses plus dependency/file ownership metadata.

2

Run preflight and stop on unrelated uncommitted changes.

3

Select the next eligible TODO phase. Sequential mode stops if any phase is already IN PROGRESS.

4

Execute only scoped work, verify locally, inspect diffs, update status, and commit only verified scoped changes.

Cron branch

1

Only enable cron when explicitly requested or when the plan already has bootstrap state/status.

2

Interactive bootstrap creates the job, records concrete ## CRON Bootstrap state, reports job identity, then stops.

3

Cron runs cannot ask questions; they re-read the plan, run preflight, execute at most allowed phases, verify, report, and exit.

4

If complete and verified, cron may pause/remove only its own job according to recorded self-stop action.

Stop conditions

  • !Unrelated uncommitted changes or destructive git actions.
  • !Branch, worktree, merge, ownership, or status-table conflicts.
  • !Failed verification without an obvious in-scope fix.
  • !Unauthorized PR, CI polling, merge/reconciliation, or cleanup behavior.
