Workflow logic

Git-Managed Plan Execution

One CRON tick claims at most one TODO phase, verifies it locally, commits a scoped checkpoint, updates the canonical plan file, then evaluates shutdown.

Core invariants

  • The plan file is canonical; the ## Phase Status table is durable state.
  • Each tick re-reads the plan and executes at most the first eligible TODO phase.
  • Changes must be scoped, locally verified, inspected, and checkpoint committed.
  • Bootstrap and shutdown state are recorded in the plan and reported.

Allowed status paths

TODOIN PROGRESSDONE DEFERREDFAILEDBLOCKEDSKIPPED
  • TODO → IN PROGRESS → DONE
  • TODO/IN PROGRESS → DEFERRED | BLOCKED
  • IN PROGRESS → FAILED
  • TODO → SKIPPED

CRON tick flow

Bootstrap gate

  1. Require schedule, repo path, plan path, branch/worktree strategy.
  2. Require verification commands, delivery target, self-stop identity/action.
  3. Create CRON job, record ## CRON Bootstrap, report, end run.

Commit gate

  1. Run configured verification.
  2. Inspect git status and diffs.
  3. Commit only scoped verified changes.
  4. Use plan:<slug> phase:<N> <keyword>.

Shutdown gate

  1. After marking a phase DONE, re-read phase status.
  2. If not all phases are DONE, report progress and wait for the next CRON tick.
  3. If all phases are DONE, run final verification and require a clean worktree.
  4. Then pause CRON by default; remove only if configured.

Hard stops

  • Missing or ambiguous input.
  • Existing IN PROGRESS phase.
  • Status table changed unexpectedly.
  • Unrelated uncommitted changes.
  • Verification failure without scoped fix.
  • User authorization required.
  • Destructive git action needed.
  • Branch/worktree/merge conflict.
  • Plan status conflict.
